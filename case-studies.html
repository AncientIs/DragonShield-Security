Financial Services Web Application & Network Penetration Testing 4 Weeks

Financial Services — Banking Platform Security Assessment

Client Background

A mid-size regional bank with over $8 billion in assets was preparing to launch a new digital banking platform. With regulatory requirements from the FFIEC and OCC mandating regular security assessments, the bank needed a comprehensive penetration test of the platform before going live.

Challenge

The bank's new digital banking platform integrated with multiple legacy core banking systems, third-party payment processors, and a mobile API layer. The complex architecture created a broad attack surface, and the bank's internal security team lacked the offensive security expertise to conduct a thorough assessment. Previous automated vulnerability scans had returned hundreds of findings with no clear prioritization, leaving the development team overwhelmed and uncertain about which issues posed genuine risk.

Approach & Methodology

DragonShield deployed a team of three senior penetration testers for a four-week engagement. We followed the OWASP Web Security Testing Guide for the web application assessment and PTES for the network infrastructure testing.

Phase 1 — Reconnaissance: We mapped the entire application architecture, identified all API endpoints, and cataloged third-party integrations.

Phase 2 — Web Application Testing: We conducted manual testing of all authentication flows, authorization controls, transaction processing logic, and data handling mechanisms. Special attention was given to business logic vulnerabilities in fund transfer and account management functions.

Phase 3 — Network Assessment: We evaluated the network segmentation between the digital banking platform, core banking systems, and the corporate network. We tested for lateral movement paths and privilege escalation opportunities.

Phase 4 — API Security Testing: We assessed the mobile API layer for authentication bypass, insecure direct object references, and data exposure vulnerabilities.

Key Findings

  • Critical: An insecure direct object reference (IDOR) vulnerability in the account API allowed authenticated users to access other customers' account details and transaction history by manipulating account identifier parameters.
  • Critical: Insufficient input validation in the internal fund transfer function allowed negative value transfers, potentially enabling unauthorized balance manipulation.
  • High: Session tokens were not invalidated server-side upon logout, allowing session replay attacks within a 30-minute window.
  • High: The administrative interface was accessible from the public internet with default credentials on a non-standard port.
  • Medium: Verbose error messages in the API disclosed internal system architecture details, database table names, and stack traces.
  • 14 additional medium and low severity findings related to security headers, TLS configuration, and cookie attributes.

Results & Impact

All critical and high-severity findings were remediated by the development team within three weeks of report delivery, with DragonShield providing remediation consultation throughout the process. Our retesting confirmed successful remediation of all findings. The digital banking platform launched on schedule, passing the subsequent regulatory security review without any findings. The bank has since engaged DragonShield for quarterly security assessments of the platform.

Key Metric: 19 vulnerabilities identified, including 2 critical findings that automated scanners had missed entirely.
Healthcare Red Team Operations 6 Weeks

Healthcare — Hospital Network Red Team Engagement

Client Background

A national healthcare provider operating 12 hospitals and over 200 outpatient clinics engaged DragonShield for a red team assessment. The organization had invested significantly in security tools and a 24/7 Security Operations Center (SOC) over the previous two years and wanted to validate the effectiveness of these investments against realistic attack scenarios.

Challenge

The healthcare provider needed to understand whether a sophisticated attacker could breach their defenses and access protected health information (PHI) governed by HIPAA. Previous penetration tests had focused on specific systems in isolation, but leadership wanted a holistic assessment that tested their people, processes, and technology together under realistic conditions. The SOC team had never been tested against a live, skilled adversary.

Approach & Methodology

DragonShield's red team conducted a six-week engagement with the objective of gaining access to systems containing PHI without being detected by the SOC. Only the CISO and General Counsel were aware of the engagement.

Week 1–2 — Reconnaissance & Social Engineering: We conducted extensive OSINT gathering, identifying key personnel, technology stacks, and organizational structure. We crafted targeted spear-phishing emails impersonating a medical equipment vendor, achieving credential harvesting from three employees.

Week 3–4 — Initial Access & Lateral Movement: Using harvested credentials, we accessed the corporate VPN and established a foothold on the internal network. We employed custom tooling to evade the organization's EDR solution and moved laterally through the network, escalating privileges via a misconfigured service account with domain admin rights.

Week 5 — Objective Achievement: We accessed a database server containing patient records in a segmented network by exploiting a trust relationship between the corporate domain and the clinical systems domain. We demonstrated access to PHI without exfiltrating any actual data.

Week 6 — Reporting & Purple Team Debrief: We compiled our findings and conducted a collaborative purple team session with the SOC, walking through the entire attack timeline and identifying detection opportunities that were missed.

Key Findings

  • The SOC did not detect the initial compromise or lateral movement for 23 days, despite multiple indicators of compromise being generated.
  • Spear-phishing emails bypassed the email security gateway due to the use of a newly registered lookalike domain.
  • A service account with domain admin privileges had not had its password rotated in over 18 months.
  • Network segmentation between corporate and clinical environments was undermined by a legacy trust relationship that had not been reviewed.
  • EDR alerts for the custom tooling were generated but classified as low priority and not investigated within the engagement window.

Results & Impact

The engagement led to a comprehensive overhaul of the organization's detection and response procedures. The SOC implemented new alert triage workflows, the IT team remediated the service account and network segmentation issues, and the organization launched an enhanced security awareness training program focused on spear-phishing recognition. A follow-up red team engagement six months later showed a 70% improvement in detection time, with the SOC identifying the initial compromise within 48 hours.

Key Metric: Mean time to detect reduced from 23 days to under 48 hours after implementing DragonShield's recommendations.
Technology / SaaS Cloud Security Testing & Web Application Testing 3 Weeks

Technology — SaaS Platform Cloud Security Assessment

Client Background

A fast-growing B2B SaaS company providing project management software to enterprise clients needed a comprehensive security assessment of their AWS-hosted platform. Several prospective enterprise customers had requested evidence of third-party security testing as part of their vendor risk assessment process, and the company was pursuing SOC 2 Type II certification.

Challenge

The SaaS platform had been built rapidly by a development team focused on feature delivery, with security considerations often deferred. The AWS environment had grown organically over three years without a formal cloud security architecture review. The company had no dedicated security team and relied on default AWS configurations for most services. They needed to identify and remediate critical security issues before their SOC 2 audit.

Approach & Methodology

DragonShield conducted a combined cloud security assessment and web application penetration test over three weeks.

Cloud Security Assessment: We reviewed the AWS environment against CIS AWS Foundations Benchmark, evaluating IAM policies, S3 bucket configurations, VPC architecture, security group rules, CloudTrail logging, and secrets management practices.

Web Application Testing: We performed a comprehensive assessment of the SaaS platform following the OWASP Testing Guide, focusing on multi-tenant isolation, authentication, authorization, and data handling.

Infrastructure as Code Review: We reviewed the company's Terraform configurations to identify security issues that would be reintroduced with each deployment.

Key Findings

  • Critical: Three S3 buckets containing customer data backups were publicly accessible due to misconfigured bucket policies.
  • Critical: A server-side request forgery (SSRF) vulnerability in the file import feature allowed access to the AWS instance metadata service, enabling retrieval of IAM role credentials.
  • High: Tenant isolation in the application relied solely on application-layer controls; a parameter manipulation vulnerability allowed cross-tenant data access.
  • High: IAM policies followed a pattern of overly permissive wildcard permissions (e.g., s3:*), violating the principle of least privilege.
  • Medium: CloudTrail logging was not enabled in three AWS regions, creating blind spots for security monitoring.
  • 22 additional findings across cloud configuration, application security, and infrastructure-as-code categories.

Results & Impact

The company remediated all critical and high-severity findings within four weeks, with DragonShield providing hands-on remediation guidance for the cloud configuration issues. The publicly accessible S3 buckets were secured immediately upon notification (within hours of discovery). The company successfully passed their SOC 2 Type II audit three months later and has since used the DragonShield assessment report to satisfy vendor security questionnaires from enterprise prospects, directly contributing to the closure of two major enterprise contracts.

Key Metric: 28 security findings identified; 3 publicly exposed S3 buckets containing customer data secured within hours of discovery.
Manufacturing Network Penetration Testing & Social Engineering 3 Weeks

Manufacturing — Industrial Control System & Corporate Network Assessment

Client Background

A multinational manufacturing company with operations across North America and Europe engaged DragonShield to assess the security of their corporate network and evaluate the risk of an attacker pivoting from the corporate environment to their operational technology (OT) network controlling industrial processes.

Challenge

The company had experienced a ransomware incident 18 months prior that disrupted production at two facilities for five days, resulting in estimated losses of $3.2 million. Since then, they had invested in network segmentation, endpoint protection, and employee security training. They needed an independent assessment to validate these improvements and identify any remaining gaps — particularly at the IT/OT boundary.

Approach & Methodology

DragonShield conducted a three-week engagement combining network penetration testing with a targeted social engineering campaign.

Week 1 — Social Engineering Campaign: We launched a phishing campaign targeting 500 employees across corporate and plant operations roles, using pretexts related to safety compliance and benefits enrollment.

Week 2 — External & Internal Network Testing: We assessed the external perimeter for vulnerabilities and then conducted internal network testing from both the corporate headquarters and a manufacturing plant location, focusing on network segmentation between IT and OT environments.

Week 3 — IT/OT Boundary Assessment: Working within carefully defined safety parameters, we evaluated the controls separating the corporate network from the OT environment, testing for paths that could allow an attacker to reach industrial control systems.

Key Findings

  • High: 12% of targeted employees clicked the phishing link, and 6% submitted corporate credentials — a significant improvement from the industry average of 15-20%, but still representing a viable initial access vector.
  • High: A legacy VPN concentrator at one plant location was running firmware with a known remote code execution vulnerability (patched version available for 14 months).
  • High: A shared service account used by an engineering application had access to both the corporate Active Directory and a jump server in the OT DMZ, creating a potential pivot path.
  • Medium: Network segmentation between IT and OT was properly implemented at the firewall level, but DNS resolution between zones was permitted, enabling potential data exfiltration via DNS tunneling.
  • Medium: Several engineering workstations in the OT environment were running unsupported operating systems with no compensating controls.

Results & Impact

The company addressed all high-severity findings within six weeks. The legacy VPN was patched immediately, the shared service account was replaced with separate accounts for each environment, and DNS resolution between IT and OT zones was restricted to specific, monitored servers. The phishing results were used to design targeted security awareness training for the departments with the highest click rates. The company now conducts annual penetration testing with DragonShield as part of their ongoing security program.

Key Metric: Potential IT-to-OT pivot path identified and eliminated; phishing susceptibility measured at 6% (below industry average).
